# Data stores & shared packages (L2)

## PostgreSQL — daily stats (`internal/dailystats`, `db/migrations`)

Access is generated by **sqlc** (`internal/dailystats/sqlc.yaml`, `sql_package: database/sql`, driver `github.com/lib/pq`). A hand-written `Repository` (`repository.go`) wraps the generated `*Queries` and does the `int`→`int32` conversions; do not edit `*.sql.go` (regenerate via `//go:generate sqlc generate` in `repository.go:9`).

The two renames matter: the DB columns, the sqlc `QueueDailyStat` struct fields (`models.go`), and the report builders all use the **new** names. Any new consumer must too.

### Queries (`queries.sql`)

- `UpsertDailyStats :exec` — `INSERT ... ON CONFLICT (queue_id, date) DO UPDATE` (idempotent per queue+day; `updated_at=NOW()`). Called by queue-monitor on → Inactive.
- `GetStatsByDateRange :many` — `WHERE queue_id=$1 AND date BETWEEN $2 AND $3 ORDER BY date ASC`. Called by queue-stats-reports. The generated params struct names the two dates `Date` and `Date_2` (`queries.sql.go`).

### Migrations (goose)

`db/migrations/0000{1,2,3}_*.sql`, goose format (`+goose Up/Down`, `StatementBegin/End`). Migration tool = `github.com/pressly/goose/v3`. In-cluster they run as the `duw-migrations` Job / image (`ghcr.io/…/duw-migrations`, base `ghcr.io/kukymbr/goose-docker:3.26.0`, `db/Dockerfile`) with `GOOSE_DRIVER=postgres`, `GOOSE_COMMAND=up`, `GOOSE_DBSTRING` from the app secret. The migrations Job manifest pins an explicit name/tag (`duw-migrations-1-1-0`), not a placeholder. sqlc reads the same migration dir as its schema source (`sqlc.yaml: schema: ../../db/migrations/`), so a new migration must land before regenerating.

Adding a column is a two-place change: a new `db/migrations/0000N_*.sql` **and** (if queried) a `queries.sql` edit, then `sqlc generate`. See `08-change-guide.md`.

## Redis — monitor state

Single key **`monitor:state`** (`internal/queuemonitor/monitorstate.go:29`), JSON-serialized `MonitorState`, TTL `STATE_TTL_SECONDS` (default 60s). Written only on graceful shutdown, read once at startup. Full mechanics in `02-queue-monitor.md`. Client: `github.com/redis/go-redis/v9`, connection from a `redis://` URL (`redis.ParseURL`). In-cluster Redis config disables persistence (`save ""`, `appendonly no`) — state is intentionally ephemeral (`06-infra.md`).

## Shared: logger (`internal/logger`)

Thin wrapper over stdlib `log/slog` with a **text handler to stdout** (`logger.go:20`). `Level` from `LOG_LEVEL` (default `info`; `debug|info|warn|error`, `config.go`). Signature note: `Error(msg string, err error, attrs ...any)` — the error is a **positional 2nd arg**, appended as `"error"` (`logger.go:37`). Several call sites pass a trailing `": "` in the message (e.g. `"Failed to forward feedback to admin: "`) — cosmetic, not a format string.

## Shared: Telegram notifier (`internal/notifications`)

`TelegramNotifier.SendMessage(ctx, chatID, text)` (`telegram.go:36`): POST to `{BaseApiUrl}/bot{BotToken}/sendMessage` with body `{chat_id, text, parse_mode:"HTML"}` (`telegram.go:30-46`). Used by all three services (queue-monitor alerts, stats reports, bot feedback forwarding). `chatID` is passed through as given — callers decide `@channel` vs numeric.

Retry (`sendMessageWithRetries`, `telegram.go:48`): `avast/retry-go/v4`, `Attempts(NOTIFICATION_TELEGRAM_MAX_RETRY_ATTEMPTS=5)`, `FixedDelay(NOTIFICATION_TELEGRAM_RETRY_DELAY_MS=500ms)`, all under `context.WithTimeout(NOTIFICATION_TELEGRAM_REQUEST_TIMEOUT_SECONDS=5s)`. Non-200 → error including the response body: `"sending message to TelegramApi failed. got unsuccessful status code: %d, api response: ..."`.

Important reliability nuance: **the request timeout (5s) bounds the whole retry loop.** A slow-but-eventually-delivered Telegram send can hit the timeout and be treated as a failure while the message was actually posted — the root of the duplicate-notifications incident (`09-reliability.md`).
